AV STUDIO
GUIA MAESTRA PARA CONSTRUIR EL EDITOR UNIVERSAL DEL PORTAFOLIO Y DE CLIENTES

ESTADO ACTUAL DE IMPLEMENTACION

- AV Studio ya tiene shell real en /admin con login Supabase
- existe un solo motor de edicion en assets/js/universal-template-editor.js
- el editor ya protege el acceso y redirige a /admin si no hay sesion valida
- ya existe vista PC / Movil dentro del editor
- ya existe modo Cliente / Pro dentro del workspace
- ya existe store compartido en assets/js/admin/content-store.js
- ya existe gestor de colecciones en /admin
- la persistencia remota usa la tabla editor_workspaces si existe
- si la tabla no existe o falla la policy, el sistema cae a guardado local sin romperse

PENDIENTE PRINCIPAL DE LA SIGUIENTE ITERACION

- conectar las colecciones universales al render publico de cada pagina
- hacer que proyectos, productos, servicios, FAQ y galeria ya alimenten bloques reales
- endurecer mas el modo cliente para esconder controles avanzados en mas superficies

====================================================================
1. PROPOSITO DEL DOCUMENTO
====================================================================

Este archivo define de manera clara, detallada y estable lo que debe
ser el editor universal del sistema.

Su objetivo es:

- evitar perder direccion durante la construccion
- alinear decisiones tecnicas y de producto
- mantener una sola vision para plantillas, implementaciones y sitios reales
- asegurar que el sistema se construya como producto vendible, no solo como demo
- servir como fuente de verdad para futuras iteraciones

Este documento describe:

- la vision del producto
- el alcance del sistema
- la experiencia esperada para clientes y administradores
- la arquitectura funcional
- los modulos del editor
- el modelo de datos base
- las fases de construccion
- criterios de calidad
- principios para no desviar el proyecto


====================================================================
2. VISION DEL PRODUCTO
====================================================================

El sistema no debe ser solo un editor visual.

Debe convertirse en un panel profesional para administrar sitios web
comerciales, contenido, colecciones, inventario, proyectos, servicios
e implementaciones reales, de forma simple para clientes y completa
para administradores.

Nombre conceptual del producto:

AV Studio

Definicion corta:

AV Studio es un sistema de administracion visual para editar y mantener
paginas web comerciales sin tocar codigo, con soporte para sitios,
colecciones, inventario, proyectos, contenido y publicacion.

Promesa comercial:

- el cliente puede actualizar su sitio sin depender de un programador
- el negocio puede gestionar contenido real desde celular o PC
- el sistema se ve profesional y transmite confianza
- el mismo editor sirve para muchas industrias
- el sistema crece con cada nuevo cliente sin rehacerse desde cero

====================================================================
4. PRINCIPIOS DEL SISTEMA
====================================================================

4.1 Un solo editor

Debe existir un solo motor de edicion.
No se deben seguir creando editores diferentes por cada pagina.

4.2 Una sola logica de acceso

La autenticacion y permisos deben seguir un esquema unico:

- login con Supabase
- permisos por sitio
- posibilidad de super admin
- acceso por /admin en implementaciones reales

4.3 Una sola forma de guardar

Todo cambio persistente debe tender a un mismo modelo:

- contenido por sitio
- contenido por pagina
- contenido por coleccion
- contenido por item

4.4 Experiencia simple para cliente

El sistema debe estar pensado para una persona no tecnica.

4.5 Experiencia completa para administrador

En PC debe existir una capa mas potente para gestion profesional.

4.6 Modo movil real

El editor debe tener vista movil propia.
No debe depender de herramientas del navegador.

4.7 El sistema debe vender

No basta con que funcione.
Debe verse serio, confiable, premium y facil de usar.

====================================================================
6. MODOS DE USO
====================================================================

6.1 Modo Cliente

Objetivo:

Permitir que un cliente haga cambios sin confundirse.

Debe priorizar:

- simplicidad
- botones claros
- lenguaje no tecnico
- cambios guiados
- pocas decisiones por pantalla

Debe permitir:

- editar textos
- cambiar imagenes
- cambiar enlaces
- agregar o quitar items simples
- cambiar orden basico
- activar o desactivar elementos
- ver como queda en movil o PC

No debe exponer:

- conceptos tecnicos innecesarios
- configuraciones avanzadas
- estructura interna del DOM
- opciones complejas sin contexto

6.2 Modo Profesional

Objetivo:

Dar control completo al administrador.

Debe permitir:

- editar estructura de pagina
- reordenar modulos
- configurar colecciones
- editar campos avanzados
- controlar estados
- manejar SEO
- manejar metadata
- configurar visibilidad
- controlar publicacion

6.3 Modo Vista

Objetivo:

Revisar resultado antes de publicar.

Debe incluir:

- vista desktop
- vista tablet si es viable
- vista movil
- opcion de navegar la pagina en esas vistas


====================================================================
7. EXPERIENCIA POR DISPOSITIVO
====================================================================

7.1 En PC

Debe ser la experiencia recomendada.

Ventajas:

- mas espacio
- paneles laterales o superiores
- manejo mas completo de colecciones
- reordenado por drag and drop
- mas visibilidad de contexto

Debe incluir:

- panel principal
- area de vista previa
- barra de herramientas
- inspector de bloque
- historial
- editor de colecciones
- biblioteca de medios

7.2 En movil

Debe ser funcional, pero mas simple.

Uso esperado:

- cambios rapidos
- revisar contenido
- corregir textos
- actualizar imagenes
- activar o desactivar items
- ver como quedo la pagina

Debe incluir:

- interfaz de panel inferior o pantalla completa
- botones grandes
- menos densidad de controles
- reordenado con subir y bajar
- flujo guiado

7.3 Conmutador de vista

Debe haber un conmutador visible para:

- PC
- tablet si aplica
- movil

Este conmutador debe vivir dentro del editor.
No debe depender de consola ni de herramientas del navegador.

16. FLUJO IDEAL PARA CLIENTE NO TECNICO
====================================================================

Flujo simple:

1. entra a /admin
2. inicia sesion
3. ve su sitio
4. elige que quiere editar
5. cambia contenido
6. revisa en PC y movil
7. guarda
8. publica

El sistema debe guiarlo con acciones claras como:

- Editar portada
- Cambiar imagen principal
- Agregar proyecto
- Subir producto
- Actualizar contacto
- Revisar vista movil
- Publicar cambios

No con lenguaje como:

- editar selector
- cambiar nodo
- editar HTML
- modificar DOM


====================================================================
17. FLUJO IDEAL PARA ADMINISTRADOR
====================================================================

Flujo profesional:

1. entra al dashboard
2. elige sitio
3. entra al editor visual o gestor de colecciones
4. reorganiza bloques o items
5. edita contenido avanzado
6. revisa responsive
7. guarda
8. publica


====================================================================
18. GUARDADO Y PERSISTENCIA
====================================================================

El sistema debe avanzar hacia un modelo central con Supabase.

Se debe guardar como minimo:

- sitio
- pagina
- modulo
- campo
- valor
- timestamp
- usuario que actualizo

El guardado local puede existir como apoyo temporal, pero no debe ser
el destino final del sistema.

Preferencia:

- Supabase como backend real
- cache local para experiencia rapida
- sincronizacion clara


====================================================================
19. PERMISOS Y SEGURIDAD
====================================================================

Debe mantenerse:

- login con Supabase
- permisos por site_id
- posibilidad de super admin
- rutas /admin protegidas

Se debe evitar:

- exponer herramientas de edicion al visitante publico
- depender solo de localStorage como seguridad
- mezclar permisos entre sitios


====================================================================

- guardar borrador

====================================================================
25. CRITERIOS DE CALIDAD
====================================================================

El sistema solo se considera correcto si:

- una plantilla se puede editar sin tocar codigo
- una implementacion se puede editar por /admin
- el login funciona con Supabase
- el cliente puede ver modo movil y modo PC
- se pueden manejar colecciones reales
- el sistema no depende de consola del navegador
- funciona en movil y PC
- es claro para usuarios no tecnicos
- no existen editores separados por pagina


====================================================================
26. ERRORES QUE NO DEBEN REPETIRSE
====================================================================

- crear paneles paralelos que duplican funciones
- mezclar varios editores con logicas distintas
- agregar accesos rotos
- editar catalogos sin sincronizar fallbacks
- hacer funciones solo para una plantilla
- pensar el sistema como demo y no como producto
- depender de hacks manuales del navegador


====================================================================
27. REGLA DE ORO
====================================================================

Cada nueva funcion debe responder esta pregunta:

"esto fortalece el editor universal o crea otra excepcion?"

Si crea otra excepcion, no se debe implementar asi.


====================================================================
28. DEFINICION FINAL DEL PRODUCTO
====================================================================

AV Studio debe terminar siendo:

un sistema profesional, visual y vendible para administrar sitios web
comerciales, plantillas e implementaciones reales, con soporte para
contenido, colecciones, inventario, proyectos, responsive preview,
publicacion y permisos por usuario.

No se construira como un conjunto de parches.
Se construira como una plataforma.
